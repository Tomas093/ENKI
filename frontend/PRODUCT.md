# Product

## Register

product

## Users

Two equally important user roles, typically in a university context:

- **Professors**: Create and host trivia sessions in class or remotely. Access the dashboard, deploy games to a smart contract, control question flow, and see final results. Tech-comfortable; familiar with MetaMask.
- **Students**: Join a live session by ID, compete in real time from desktop (primary) or mobile (supported), and claim on-chain prizes and diploma NFTs. They should feel zero blockchain friction — the Web3 layer must be invisible. Most students are not crypto-native.

Context of use: active class sessions and homework. High time pressure. Mobile must work but desktop is primary.

## Product Purpose

ENKI is a Web3-native trivia platform for university courses. Professors stake a prize pool; students pay an entry fee and compete for on-chain rewards and diploma NFTs. The smart contract enforces fairness — no professor can manipulate results.

Success looks like: a professor creates a game in under 60 seconds, students join in under 30 seconds without touching MetaMask jargon, and the session runs without technical friction from start to diploma claim.

## Brand Personality

**Competitive. Tangible. Alive.**

ENKI should feel like a game show with real stakes, not a corporate tool. The energy is arcade — focused, alive, slightly electric. Rewards feel real because they are real (ETH + NFTs). The interface never apologizes for being on-chain; it just makes it effortless.

- Voice: direct, active, confident. No "seamless" or "revolutionizing". Say "compete", "stake", "win", "claim".
- Tone: energetic but never chaotic. Focused excitement. Think: live sports scoreboard meets clean product dashboard.
- 3-word personality: **Competitive. Tangible. Alive.**

## Anti-references

- **Kahoot**: too infantile. Primary colors, confetti spam, baby font sizing. ENKI is for adults.
- **Duolingo**: mascot-dependent gamification feels hollow when the rewards are real ETH.
- **Raw MetaMask / crypto UI**: techno-jargon everywhere, 0x addresses exposed, dark terminal aesthetic. Alienates students.
- **Menti / Slido**: corporate blandness. Pastel grays, zero personality, feels like a meeting tool.

## Design Principles

1. **The Web3 layer is invisible to students.** Wallet addresses, transaction hashes, gas — none of this belongs in the student UI. The game IS the experience.
2. **Speed is a feature.** Every second a student spends reading UI is a second they're not competing. Hierarchy must be instant; the primary action must be obvious without scanning.
3. **Stakes are real, so the UI must feel real.** Cheap aesthetics undermine trust. If there's ETH on the line, the interface should feel like it belongs there — not a toy, not a prototype.
4. **Two products share one shell.** Professor and student views are fundamentally different workflows. Don't compromise either to share components. Diverge where it helps.
5. **Motion earns its place.** Waiting rooms and leaderboards are high-tension moments. Purposeful motion — live player counts updating, scores cascading — amplifies that tension. Gratuitous animation kills focus.

## Accessibility & Inclusion

- Responsive design: desktop primary, mobile supported (students joining from phones).
- No specific WCAG level requirement, but contrast must be readable in all ambient lighting (classrooms vary).
- No reduced-motion exemptions explicitly required, but animations should not be the only conveyor of state changes.
